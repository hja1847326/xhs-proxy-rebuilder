# 扩展路模型 V1：VLAN + netns + gost

## 结论

针对华为云 `4+N` 模型里的 `N`（来自 `ip_origin.txt` 的扩展路），正确落地方式不是把扩展 IP 直接旁挂到主命名空间的 `eth0` / `eth1`，而是：

1. 基于主设备创建 `802.1Q VLAN` 子接口
2. 每条扩展路创建一个独立 `netns`
3. 把对应 VLAN 子接口移入该 `netns`
4. 在该 `netns` 内配置：
   - 扩展 IP
   - 默认路由
   - `resolv.conf`
5. 在该 `netns` 内启动独立代理进程（当前对标旧方案为 `gost`）

---

## 为什么不能继续走“主空间旁挂”

在真实新机器 `ecs-99bc` 上，对第 5 路：

- `122.9.187.215 -> 192.168.0.88:19005` 的 SYN 能进机器
- 把 `192.168.0.88` 直接挂到 `eth0` / `eth1` 后，本机可监听
- 但外部无法形成完整可用代理

这说明扩展路不是简单的“主空间补一个地址 + 补几条规则”模型。

---

## 成功机只读反推结果

在一台已运行旧脚本、可交付 10 条 SK5 的机器上，只读确认到：

### 输入样本
`/root/ip_origin.txt` 中每行形如：

```text
192.168.0.54  fa:16:3e:f2:4a:80 3783
192.168.0.163 fa:16:3e:f2:4a:ed 681
...
```

### 命名空间
系统中存在对应的：

```text
ns3783
ns681
ns202
ns2120
ns1727
ns1867
```

### netns 内接口
每个 `netns` 中的扩展接口都是 `802.1Q VLAN` 子接口，例如：

```text
eth0.3783  vlan id 3783
eth0.681   vlan id 681
eth0.202   vlan id 202
```

### netns 内网络
每个 `netns` 内都具备：

- 独立 IP（例如 `192.168.0.54/24`）
- 默认路由（例如 `default via 192.168.0.1 dev eth0.3783`）
- 独立 DNS（`/etc/netns/nsXXXX/resolv.conf`）
- 独立 `gost` 监听（统一复用 `18848`，但命名空间不同，因此不冲突）

## 新机第 5 路实测

### 目标

```text
bind_ip: 192.168.0.88
vlan: 1866
source_mac: fa:16:3e:8b:e3:3b
public_ip: 122.9.187.215
```

### 正确试验模型

在新机上按下列结构临时落地：

- `eth0.1866`
- `ns1866`
- `192.168.0.88/24`
- `default via 192.168.0.1 dev eth0.1866`
- `gost :19005`
- `/etc/netns/ns1866/resolv.conf` 使用 `223.5.5.5`

### 实测结论

- `ns1866` 内 `ping 192.168.0.1` 成功
- `ns1866` 内 `ping 1.1.1.1` 成功
- `ns1866` 内 `curl http://ifconfig.me/ip` 返回：

```text
122.9.187.215
```

这表示第 5 路出口已经真实落到目标公网 IP。

### 代理层结论

- SOCKS5 握手 / 认证成功
- 对目标 IP 的 CONNECT 成功
- 行为特征与成功机扩展路一致
- `api.ipify.org` 并不能作为唯一成功标准；在成功机扩展 ns 内同样可能失败
- `ifconfig.me/ip` 返回正确公网 IP 才是更可靠的验证标准

## V1 实施建议

### 基础 4 路
继续保留现有模型：

- 主 IP
- 2 个 VIP
- 1 个固定辅助网卡 / table 100

### txt 扩展 N 路
对每一条 `ip_origin` 扩展项：

1. 取 `bind_ip`
2. 取 `vlan_id`
3. 创建 `eth0.<vlan_id>`
4. 创建 `ns<vlan_id>`
5. 把 `eth0.<vlan_id>` 移入 `ns<vlan_id>`
6. 在 `ns<vlan_id>` 内配置：
   - `bind_ip/24`
   - `default via 192.168.0.1`
   - `resolv.conf`
7. 在 `ns<vlan_id>` 内启动独立 `gost`
8. 导出对应代理条目
